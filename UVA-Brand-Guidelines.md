## Typography

### Font Hierarchy

The University uses a three-tiered font system:

1. **Primary Font:** Franklin Gothic (sans serif)
2. **Secondary Font:** Bodoni (serif accent)
3. **Logo Font:** Adobe Caslon (serif)

---

### 1. Adobe Caslon

**Type:** Serif
**Primary Use:** University logo system, formal applications

- Historical significance: The font used by John Dunlap to typeset Thomas Jefferson's Declaration of Independence
- Best suited for: letterhead, business cards, envelopes, and formal materials
- Conveys: Traditional, scholarly authority

**Alternative Matches:**
- Libre Caslon Text (Google Fonts)
- Garamond

---

### 2. Franklin Gothic

**Type:** Sans Serif
**Primary Use:** Headlines and body copy

- Most flexible font in the UVA brand
- Modern, clean, and versatile
- Available in both non-condensed and condensed/compressed variants
- All weights and faces available but should be used with discretion

**Usage:**
- Headlines: Use bold or heavy weights
- Body copy: Use regular or book weights
- Compressed variants: Use when space is limited

**Alternative Matches:**
- Libre Franklin (Google Fonts)
- Helvetica
- Arial

---

### 3. Bodoni

**Type:** Serif
**Primary Use:** Accent design elements

- **Bodoni Poster Italic** is most frequently used
- Provides a bold but studious aesthetic
- Should complement, not replace, primary fonts
- Use sparingly for visual impact

**Alternative Matches:**
- Bodoni Moda (Google Fonts)
- Palatino

## Accessibility

The University of Virginia is committed to creating accessible communications that comply with **WCAG (Web Content Accessibility Guidelines) 2.1 Level AA** standards.

### Understanding WCAG 2.1

WCAG 2.1 is built on four foundational principles, often remembered by the acronym **POUR**:

1. **Perceivable** - Information and user interface components must be presentable to users in ways they can perceive
2. **Operable** - User interface components and navigation must be operable
3. **Understandable** - Information and the operation of user interface must be understandable
4. **Robust** - Content must be robust enough to be interpreted by a wide variety of user agents, including assistive technologies

### Conformance Levels

- **Level A:** Minimum level of conformance (essential accessibility features)
- **Level AA:** Mid-range level addressing major barriers (recommended standard)
- **Level AAA:** Highest level of conformance (enhanced accessibility)

**UVA Standard:** WCAG 2.1 Level AA (minimum)
**Recommended:** WCAG 2.1 Level AAA when possible

---

## WCAG 2.1 Level AA Complete Compliance Checklist

This comprehensive checklist includes all Level A and Level AA success criteria required for WCAG 2.1 AA conformance.

---

### Principle 1: Perceivable

Information and user interface components must be presentable to users in ways they can perceive.

#### Guideline 1.1 - Text Alternatives

**1.1.1 Non-text Content (Level A)**
- ‚úÖ Provide text alternatives (alt text) for all non-text content
- ‚úÖ Use empty alt text (`alt=""`) for decorative images
- ‚úÖ Provide descriptive alt text for informative images
- ‚úÖ Ensure linked images have descriptive alternative text
- ‚úÖ Provide alternative options (audio/OTP) for CAPTCHA
- ‚úÖ Use proper ARIA labels for icon buttons and controls

**Implementation:**
```html
<!-- Informative image -->
<img src="uva-logo.png" alt="University of Virginia logo">

<!-- Decorative image -->
<img src="decorative-line.png" alt="">

<!-- Linked image -->
<a href="/home">
  <img src="home-icon.png" alt="Return to homepage">
</a>
```

---

#### Guideline 1.2 - Time-based Media

**1.2.1 Audio-only and Video-only (Prerecorded) (Level A)**
- ‚úÖ Provide text transcripts for prerecorded audio-only content
- ‚úÖ Provide text alternatives or audio descriptions for video-only content
- ‚úÖ Make transcripts easy to find and access

**1.2.2 Captions (Prerecorded) (Level A)**
- ‚úÖ Provide synchronized captions for all prerecorded video with audio
- ‚úÖ Ensure captions are accurate and complete
- ‚úÖ Include speaker identification when needed
- ‚úÖ Describe relevant sound effects in captions

**1.2.3 Audio Description or Media Alternative (Prerecorded) (Level A)**
- ‚úÖ Provide audio descriptions OR a full text alternative for prerecorded video
- ‚úÖ Describe important visual information not conveyed in audio

**1.2.4 Captions (Live) (Level AA)**
- ‚úÖ Provide captions for all live audio content in synchronized media
- ‚úÖ Include real-time captions for live broadcasts
- ‚úÖ Ensure live captions are as accurate as possible

**1.2.5 Audio Description (Prerecorded) (Level AA)**
- ‚úÖ Provide audio descriptions for all prerecorded video content
- ‚úÖ Describe visual information necessary to understand content
- ‚úÖ Integrate descriptions during natural pauses in dialogue

---

#### Guideline 1.3 - Adaptable

**1.3.1 Info and Relationships (Level A)**
- ‚úÖ Use semantic HTML markup (headings, lists, tables, forms)
- ‚úÖ Associate form labels with their inputs using `<label>` tags
- ‚úÖ Use `<th>` and proper table structure for data tables
- ‚úÖ Mark up lists with `<ul>`, `<ol>`, or `<dl>` tags
- ‚úÖ Use ARIA landmarks for page regions
- ‚úÖ Ensure programmatic determination of structure and relationships

**Implementation:**
```html
<!-- Semantic headings -->
<h1>Course Title</h1>
<h2>Section 1: Introduction</h2>
<h3>Subsection 1.1</h3>

<!-- Proper form labels -->
<label for="email">Email Address:</label>
<input type="email" id="email" name="email">

<!-- Data table structure -->
<table>
  <thead>
    <tr>
      <th>Assignment</th>
      <th>Due Date</th>
      <th>Points</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Essay 1</td>
      <td>Oct 15</td>
      <td>100</td>
    </tr>
  </tbody>
</table>
```

**1.3.2 Meaningful Sequence (Level A)**
- ‚úÖ Ensure reading and navigation order is logical and intuitive
- ‚úÖ Present content in a meaningful sequence
- ‚úÖ Ensure DOM order matches visual order
- ‚úÖ Test with screen readers to verify logical flow

**1.3.3 Sensory Characteristics (Level A)**
- ‚úÖ Don't rely solely on shape, size, visual location, orientation, or sound
- ‚úÖ Provide visible labels for all controls
- ‚úÖ Use multiple cues (not just "click the round button")
- ‚úÖ Complement visual instructions with text

**Example:**
- ‚ùå "Click the button on the right to continue"
- ‚úÖ "Click the 'Continue' button to proceed"

**1.3.4 Orientation (Level AA)**
- ‚úÖ Allow content to be viewed in both portrait and landscape orientations
- ‚úÖ Don't restrict content to a single display orientation
- ‚úÖ Use responsive design techniques
- ‚úÖ Exception: When specific orientation is essential (e.g., bank check, piano app)

**1.3.5 Identify Input Purpose (Level AA)**
- ‚úÖ Use autocomplete attributes for common input fields
- ‚úÖ Help browsers autofill forms correctly
- ‚úÖ Use HTML5 input types appropriately

**Implementation:**
```html
<input type="email" name="email" autocomplete="email">
<input type="tel" name="phone" autocomplete="tel">
<input type="text" name="fname" autocomplete="given-name">
```

---

#### Guideline 1.4 - Distinguishable

**1.4.1 Use of Color (Level A)**
- ‚úÖ Don't use color as the only means of conveying information
- ‚úÖ Use additional indicators (icons, patterns, text labels)
- ‚úÖ Ensure links are distinguishable from surrounding text
- ‚úÖ Provide non-color cues for required fields, errors, etc.

**Example:**
- ‚ùå Required fields shown only in red
- ‚úÖ Required fields marked with asterisk (*) AND color

**1.4.2 Audio Control (Level A)**
- ‚úÖ Provide mechanism to pause, stop, or control volume for auto-playing audio
- ‚úÖ Don't auto-play audio for more than 3 seconds
- ‚úÖ Include visible controls for all audio/video

**1.4.3 Contrast (Minimum) (Level AA)**
- ‚úÖ Normal text: 4.5:1 contrast ratio minimum
- ‚úÖ Large text (18.66px bold or 24px+): 3:1 contrast ratio minimum
- ‚úÖ Test all text/background combinations
- ‚úÖ Ensure UI components have 3:1 contrast against adjacent colors

**UVA-Compliant Combinations:**
- UVA Blue (#232D4B) on White (#FFFFFF) = 12.4:1 ‚úì
- UVA Orange (#E57200) on White (#FFFFFF) = 4.6:1 ‚úì
- UVA Blue (#232D4B) on Yellow (#FDDA24) = 11.8:1 ‚úì
- Text Gray (#666666) on White (#FFFFFF) = 5.7:1 ‚úì

**Testing Tools:**
- WebAIM Contrast Checker
- Color Contrast Analyzer (CCA)
- Browser DevTools accessibility features

**1.4.4 Resize Text (Level AA)**
- ‚úÖ Allow text to be resized up to 200% without loss of content or functionality
- ‚úÖ Use relative units (em, rem, %) instead of fixed pixels
- ‚úÖ Ensure no horizontal scrolling when zoomed
- ‚úÖ Test at 200% browser zoom
- ‚úÖ Ensure layouts don't break at larger text sizes

**1.4.5 Images of Text (Level AA)**
- ‚úÖ Use actual HTML/CSS text instead of text images whenever possible
- ‚úÖ Only use text images when essential (logos, specific presentation)
- ‚úÖ Ensure customizable alternative when text images are used

**Example:**
- ‚ùå Bitmap image of stylized heading
- ‚úÖ CSS-styled HTML heading with custom fonts

**1.4.10 Reflow (Level AA)**
- ‚úÖ Support content presentation without 2D scrolling at:
  - 320 CSS pixels width (vertical scrolling content)
  - 256 CSS pixels height (horizontal scrolling content)
- ‚úÖ Use responsive web design from conception
- ‚úÖ Ensure content reflows at 400% zoom
- ‚úÖ Avoid fixed-width containers
- ‚úÖ Test on mobile viewports (320px wide)

**Exception:** Content requiring 2D layout (maps, diagrams, data tables, toolbars)

**1.4.11 Non-text Contrast (Level AA)**
- ‚úÖ UI components: 3:1 contrast ratio against adjacent colors
- ‚úÖ Graphical objects: 3:1 contrast ratio
- ‚úÖ Focus indicators: 3:1 contrast ratio
- ‚úÖ Form input borders: 3:1 contrast ratio
- ‚úÖ Button boundaries: 3:1 contrast ratio

**Examples:**
- Text input borders against background
- Icon buttons against their background
- Chart elements and data visualizations
- Focus rings and selection indicators

**1.4.12 Text Spacing (Level AA)**
- ‚úÖ Support user-adjusted text spacing without content loss:
  - Line height (line spacing) at least 1.5x font size
  - Paragraph spacing at least 2x font size
  - Letter spacing at least 0.12x font size
  - Word spacing at least 0.16x font size
- ‚úÖ Don't restrict or override user stylesheet preferences
- ‚úÖ Test with browser text spacing adjustments

**1.4.13 Content on Hover or Focus (Level AA)**
- ‚úÖ Additional content triggered by hover/focus must be:
  1. **Dismissible:** Can be dismissed without moving focus (ESC key)
  2. **Hoverable:** Pointer can move over new content without it disappearing
  3. **Persistent:** Remains visible until dismissed or no longer relevant
- ‚úÖ Apply to tooltips, dropdown menus, pop-ups, etc.

### Principle 2: Operable

User interface components and navigation must be operable.

#### Guideline 2.1 - Keyboard Accessible

**2.1.1 Keyboard (Level A)**
- ‚úÖ All functionality available via keyboard
- ‚úÖ No keyboard traps
- ‚úÖ Ensure custom widgets are keyboard accessible
- ‚úÖ Support standard keyboard navigation (Tab, Enter, Space, Arrow keys)
- ‚úÖ Test entire site using only keyboard

**2.1.2 No Keyboard Trap (Level A)**
- ‚úÖ Users can navigate away from any component using keyboard alone
- ‚úÖ If custom keys required, provide clear instructions
- ‚úÖ Ensure modals and dialogs can be closed with ESC key
- ‚úÖ Test navigation into and out of all interactive elements

**2.1.4 Character Key Shortcuts (Level A)**
- ‚úÖ Keyboard shortcuts using only character keys must:
  1. Be able to be turned off, OR
  2. Be able to be remapped, OR
  3. Only be active when component has focus
- ‚úÖ Prevent conflicts with assistive technology shortcuts
- ‚úÖ Provide settings to customize shortcuts

---

#### Guideline 2.2 - Enough Time

**2.2.1 Timing Adjustable (Level A)**
- ‚úÖ Provide options to turn off, adjust, or extend time limits
- ‚úÖ Warn users before time expires
- ‚úÖ Allow at least 20 seconds to extend timeout
- ‚úÖ Exceptions: Real-time events, essential timers, >20 hour limits

**2.2.2 Pause, Stop, Hide (Level A)**
- ‚úÖ Provide controls to pause, stop, or hide moving/blinking content
- ‚úÖ Apply to auto-updating content (news tickers, carousels)
- ‚úÖ Apply to moving, blinking, or scrolling content lasting >5 seconds
- ‚úÖ Exception: When movement is essential

---

#### Guideline 2.3 - Seizures and Physical Reactions

**2.3.1 Three Flashes or Below Threshold (Level A)**
- ‚úÖ Don't include content that flashes more than 3 times per second
- ‚úÖ Keep flashing below general flash and red flash thresholds
- ‚úÖ Avoid strobing effects
- ‚úÖ Test animations and transitions

---

#### Guideline 2.4 - Navigable

**2.4.1 Bypass Blocks (Level A)**
- ‚úÖ Provide "skip to main content" link
- ‚úÖ Use ARIA landmarks (navigation, main, complementary)
- ‚úÖ Use proper heading structure
- ‚úÖ Allow users to bypass repetitive content

**Implementation:**
```html
<a href="#main-content" class="skip-link">Skip to main content</a>
<nav aria-label="Main navigation">...</nav>
<main id="main-content">...</main>
```

**2.4.2 Page Titled (Level A)**
- ‚úÖ Provide descriptive and informative page titles
- ‚úÖ Use unique titles for each page
- ‚úÖ Place most specific information first
- ‚úÖ Format: Page Name - Section - Site Name

**Example:**
```html
<title>Assignment 1: Research Paper - HIST 101 - UVA</title>
```

**2.4.3 Focus Order (Level A)**
- ‚úÖ Ensure logical and intuitive focus order
- ‚úÖ Match focus order to visual order when possible
- ‚úÖ Test keyboard navigation flow
- ‚úÖ Ensure no unexpected focus changes

**2.4.4 Link Purpose (In Context) (Level A)**
- ‚úÖ Make link text descriptive and meaningful
- ‚úÖ Avoid generic "click here" or "read more"
- ‚úÖ Ensure link purpose is clear from link text or context
- ‚úÖ Use aria-label or aria-labelledby when needed

**Example:**
- ‚ùå "Click here for the syllabus"
- ‚úÖ "Download HIST 101 syllabus (PDF)"

**2.4.5 Multiple Ways (Level AA)**
- ‚úÖ Provide multiple ways to locate pages (navigation menu, search, sitemap)
- ‚úÖ Include site-wide navigation
- ‚úÖ Provide search functionality
- ‚úÖ Offer breadcrumb navigation
- ‚úÖ Exception: Pages that are steps in a process

**2.4.6 Headings and Labels (Level AA)**
- ‚úÖ Use descriptive headings for all sections
- ‚úÖ Ensure labels clearly describe purpose
- ‚úÖ Make headings informative and concise
- ‚úÖ Use proper heading hierarchy (H1 ‚Üí H2 ‚Üí H3)

**2.4.7 Focus Visible (Level AA)**
- ‚úÖ Ensure keyboard focus is always clearly visible
- ‚úÖ Don't remove default focus indicators without replacement
- ‚úÖ Use high-contrast focus indicators
- ‚úÖ Test focus visibility on all interactive elements

**CSS Example:**
```css
/* Enhance default focus outline */
:focus {
  outline: 3px solid #E57200; /* UVA Orange */
  outline-offset: 2px;
}

/* Never remove focus without replacement */
:focus:not(:focus-visible) {
  outline: none; /* Only if providing custom indicator */
}
```

#### Guideline 2.5 - Input Modalities

**2.5.1 Pointer Gestures (Level A)**
- ‚úÖ All multipoint or path-based gestures must have single-pointer alternative
- ‚úÖ Avoid requiring complex gestures (pinch-zoom, swipe patterns)
- ‚úÖ Provide simple tap/click alternatives
- ‚úÖ Exception: When multipoint gesture is essential

**2.5.2 Pointer Cancellation (Level A)**
- ‚úÖ Prevent accidental activation
- ‚úÖ Complete actions on "up" event (mouse up, touch end)
- ‚úÖ Allow users to abort or undo actions
- ‚úÖ Confirm before completing irreversible actions

**2.5.3 Label in Name (Level A)**
- ‚úÖ Ensure visible text labels match accessible names
- ‚úÖ Include visible label text in programmatic name
- ‚úÖ Support voice control users

**Example:**
```html
<!-- Visible label matches accessible name -->
<button aria-label="Submit application form">Submit</button>
```

**2.5.4 Motion Actuation (Level A)**
- ‚úÖ Provide alternatives to device motion or user motion
- ‚úÖ Allow users to disable motion-activated features
- ‚úÖ Don't require shaking, tilting, or gesturing
- ‚úÖ Provide button/keyboard alternatives

### Principle 3: Understandable

Information and the operation of user interface must be understandable.

#### Guideline 3.1 - Readable

**3.1.1 Language of Page (Level A)**
- ‚úÖ Specify the language of each page using `lang` attribute
- ‚úÖ Helps screen readers pronounce content correctly
- ‚úÖ Assists translation tools

**Implementation:**
```html
<html lang="en">
```

**3.1.2 Language of Parts (Level AA)**
- ‚úÖ Identify language changes within content
- ‚úÖ Apply lang attribute to foreign language passages
- ‚úÖ Helps screen readers switch pronunciation

**Example:**
```html
<p>The course motto is <span lang="la">Veritas et utilitas</span> (Truth and utility).</p>
```

---

#### Guideline 3.2 - Predictable

**3.2.1 On Focus (Level A)**
- ‚úÖ Don't trigger unexpected context changes on focus
- ‚úÖ Avoid auto-submitting forms when field receives focus
- ‚úÖ Don't launch new windows on focus
- ‚úÖ Keep focus behavior predictable

**3.2.2 On Input (Level A)**
- ‚úÖ Don't trigger unexpected context changes on input
- ‚úÖ Warn users before auto-submitting forms
- ‚úÖ Provide submit button rather than onChange submission
- ‚úÖ Make behavior predictable

**3.2.3 Consistent Navigation (Level AA)**
- ‚úÖ Keep navigation menus in same location across pages
- ‚úÖ Use consistent navigation order
- ‚úÖ Maintain predictable navigation patterns
- ‚úÖ Users should know where to find navigation

**3.2.4 Consistent Identification (Level AA)**
- ‚úÖ Use consistent labels for same functionality
- ‚úÖ Keep icons and buttons consistent across site
- ‚úÖ Don't change terminology for same features
- ‚úÖ If icon means "download" on one page, use same icon throughout

---

#### Guideline 3.3 - Input Assistance

**3.3.1 Error Identification (Level A)**
- ‚úÖ Clearly identify input errors
- ‚úÖ Describe errors in text
- ‚úÖ Don't rely solely on color to indicate errors
- ‚úÖ Provide specific error messages

**Example:**
- ‚ùå Red border only
- ‚úÖ "Error: Email address is required. Please enter a valid email address."

**3.3.2 Labels or Instructions (Level A)**
- ‚úÖ Provide labels for all form inputs
- ‚úÖ Offer instructions for complex inputs
- ‚úÖ Indicate required fields clearly
- ‚úÖ Provide format examples when needed

**Example:**
```html
<label for="phone">Phone Number (format: 555-123-4567)</label>
<input type="tel" id="phone" required aria-required="true">
```

**3.3.3 Error Suggestion (Level AA)**
- ‚úÖ Suggest corrections for input errors
- ‚úÖ Provide helpful error messages
- ‚úÖ Offer examples of correct format
- ‚úÖ Don't just say "invalid" - explain why and how to fix

**Example:**
- ‚ùå "Invalid date"
- ‚úÖ "Invalid date format. Please use MM/DD/YYYY (e.g., 10/08/2025)"

**3.3.4 Error Prevention (Legal, Financial, Data) (Level AA)**
- ‚úÖ For legal/financial transactions, provide one of:
  1. Reversible submissions
  2. Data validation with correction opportunity
  3. Confirmation page before final submission
- ‚úÖ Apply to any commitment that creates legal obligations
- ‚úÖ Use confirmation dialogs for irreversible actions

**Example:**
```html
<!-- Review before submit -->
<h2>Review Your Submission</h2>
<p>Please review your information before submitting.</p>
<button>Edit</button>
<button>Confirm and Submit</button>
```

### Principle 4: Robust

Content must be robust enough to be interpreted by a wide variety of user agents, including assistive technologies.

#### Guideline 4.1 - Compatible

**4.1.1 Parsing (Level A)**
- ‚úÖ Use valid HTML (no duplicate IDs, proper nesting, complete tags)
- ‚úÖ Ensure clean, well-formed markup
- ‚úÖ Validate HTML with W3C validator
- ‚úÖ Avoid markup errors that confuse assistive technologies

**4.1.2 Name, Role, Value (Level A)**
- ‚úÖ Ensure all UI components have programmatically determined:
  - **Name:** What it's called
  - **Role:** What it is (button, link, checkbox, etc.)
  - **Value/State:** Current state (checked, expanded, etc.)
- ‚úÖ Use semantic HTML elements when possible
- ‚úÖ Use ARIA appropriately for custom widgets
- ‚úÖ Ensure state changes are announced to assistive technologies

**Implementation:**
```html
<!-- Native HTML (preferred) -->
<button>Submit</button>

<!-- Custom widget with ARIA -->
<div role="button" tabindex="0" aria-pressed="false">
  Toggle
</div>

<!-- Form input -->
<label for="username">Username</label>
<input type="text" id="username" aria-required="true">
```

**4.1.3 Status Messages (Level AA)**
- ‚úÖ Announce status messages to screen readers
- ‚úÖ Use ARIA live regions for dynamic content updates
- ‚úÖ Don't require focus change to perceive status
- ‚úÖ Apply to success messages, progress indicators, errors

**Implementation:**
```html
<!-- Success message -->
<div role="status" aria-live="polite">
  Your assignment has been submitted successfully.
</div>

<!-- Error alert -->
<div role="alert" aria-live="assertive">
  Error: Please correct the highlighted fields.
</div>

<!-- Progress indicator -->
<div role="status" aria-live="polite" aria-atomic="true">
  Uploading: 45% complete
</div>
```

## Typography Accessibility Guidelines

### Font Size Requirements

- **Minimum body text:** 16px (1rem)
- **Large text threshold:** 18.66px bold (14pt) or 24px regular (18pt)
- **Headings:** Scale appropriately (H1 largest ‚Üí H6 smallest)

### Line Height & Spacing

- **Line height:** Minimum 1.5x font size (1.5em or 150%)
- **Paragraph spacing:** Minimum 2x font size below paragraphs
- **Letter spacing:** Minimum 0.12x font size
- **Word spacing:** Minimum 0.16x font size

### Line Length

- **Optimal:** 50-75 characters per line
- **Maximum:** 80 characters per line for readability

### Implementation Example

```css
body {
  font-family: 'Franklin Gothic', Arial, sans-serif;
  font-size: 16px; /* 1rem */
  line-height: 1.5; /* 24px */
  color: #232D4B; /* UVA Blue */
}

p {
  margin-bottom: 1.5em; /* 2x font size */
  max-width: 75ch; /* Optimal line length */
}

h1 {
  font-size: 2.5rem; /* 40px */
  line-height: 1.2;
  margin-bottom: 0.5em;
}
```

## Common Accessibility Mistakes to Avoid

### ‚ùå Don't Do This

1. **Missing alt text:** `<img src="logo.png">`
2. **Empty links:** `<a href="#"></a>`
3. **Generic link text:** `<a href="#">Click here</a>`
4. **Unlabeled form inputs:** `<input type="text">`
5. **Removing focus outlines:** `*:focus { outline: none; }`
6. **Color-only indicators:** Red = error, green = success (no text)
7. **Inaccessible dropdowns:** Custom JS without keyboard support
8. **Auto-playing video:** With sound, no controls
9. **Flashing content:** More than 3 flashes per second
10. **Fixed text size:** Using px units exclusively
11. **Skipping heading levels:** H1 ‚Üí H3 (skipping H2)
12. **Placeholders as labels:** No `<label>` element
13. **Opening new windows without warning**
14. **Keyboard traps:** Can't tab out of modal
15. **Low contrast text:** Gray on gray

### ‚úÖ Do This Instead

1. **Descriptive alt text:** `<img src="logo.png" alt="University of Virginia logo">`
2. **Meaningful links:** `<a href="/syllabus">Download course syllabus (PDF, 2MB)</a>`
3. **Labeled inputs:** `<label for="email">Email:</label><input type="email" id="email">`
4. **Visible focus:** Custom high-contrast focus indicators
5. **Multiple cues:** Color + icon + text for errors
6. **Semantic HTML:** Use `<button>`, `<nav>`, `<main>`, etc.
7. **Keyboard accessible:** Test all interactions with Tab, Enter, Space
8. **User control:** Pause/stop/hide for auto-playing content
9. **Proper headings:** H1 ‚Üí H2 ‚Üí H3 in order
10. **Responsive text:** Use rem/em units
11. **ARIA when needed:** For custom widgets
12. **Focus management:** Return focus after modal closes

## Summary: WCAG 2.1 Level AA Compliance

To achieve WCAG 2.1 Level AA compliance, you must meet:

- ‚úÖ **All 30 Level A success criteria**
- ‚úÖ **All 20 Level AA success criteria**
- ‚úÖ **Total: 50 success criteria**

### The Four Principles (POUR)

1. **Perceivable:** Users can perceive the information being presented
2. **Operable:** Users can operate the interface
3. **Understandable:** Users can understand the information and interface
4. **Robust:** Content works with current and future technologies

### Key Takeaways

- Test with real users, including those using assistive technologies
- Accessibility benefits everyone, not just users with disabilities
- Build accessibility in from the start, not as an afterthought
- Use semantic HTML and native elements when possible
- Test early and test often
- Involve users with disabilities in testing
- Stay current with evolving standards

**Remember:** Accessibility is not a checklist‚Äîit's an ongoing commitment to inclusive design.
